# Step 14. 기능 우선순위 분류 (임팩트/시간)

> 목적: `docs/11`의 **페이지별 기능(F-xx)** 마다 **1~4순위**를 정하고, 무엇을 만들고·버릴지·mock 할지 확정한다.

> **Guide:**  
> 1. `docs/11_page_routing_split.md`의 **페이지별 기능 목록**을 본다.  
> 2. **기능 하나마다** 아래 4가지 중 하나에 넣는다. (페이지 단위가 아니라 **기능 단위**)  
> 3. 팀이 토론하며 **1순위 = 이번에 꼭 구현**할 것을 맞춘다.  
> - **1순위** — 높은 임팩트 / 짧은 구현 시간 (즉시 개발)  
> - **2순위** — 높은 임팩트 / 긴 구현 시간 (핵심 확장)  
> - **3순위** — 낮은 임팩트 / 짧은 구현 시간 (여유 있으면)  
> - **4순위** — 낮은 임팩트 / 긴 구현 시간 (이번 범위 **스펙 아웃**·mock 후보)

**프롬프트 예시**

```
docs/11_page_routing_split.md만 읽고 docs/14_priority_matrix.md를 채워줘.

규칙:
- Step 11의 모든 F-xx를 1~4순위 중 하나에 반드시 배치 (누락 없음)
- 항목 형식: [ ] **F-xx** (기능명) — (페이지명)
- 해커톤 MVP: 핵심 기능 위주로 1순위 배치, 부가 기능은 3~4순위
- "Step 11 기능 (분류 전)" 섹션은 참고용이면 팀이 채운 그대로 두거나, Step 11과 동일하게 유지
- 구조·Guide·Tip은 유지
```

> **Tip — 읽어보기**

**서비스의 핵심 기능만 구현하기**  
예: 친구와 함께 노래 듣기가 핵심이면, **그 기능만** 제대로 구현한다.

**서비스의 핵심이 아닌 기능은 과감히 버리기**  
아래는 보통 핵심이 아니다. 과감히 빼거나, **동작하지 않는 mock**으로 둔다.

- *현실적 운영용:* 회원가입, 로그인, 로그아웃, 비밀번호 변경 등 — **데모에 필수 아님**
- *있으면 좋은 기능:* 플레이리스트 커스텀, 프로필 꾸미기, 소셜 공유 등 — **시간 많이 남을 때만**

**미구현 기능에는 ‘준비중입니다.’ 알림**  
- 버튼을 눌렀을 때 **아무 반응 없음** → 버그처럼 보임  
- **‘준비중입니다.’** 표시 → 의도된 미구현으로 보임  
→ 4순위·mock 기능도 **후속 UX**는 챙긴다.
